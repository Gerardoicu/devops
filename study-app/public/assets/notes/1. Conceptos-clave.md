## 1. Servicios que debes distinguir sin dudar

### AWS Config

Que es:

- servicio de compliance y evaluacion de configuracion
- responde "cumple o no cumple"

Cuando suele ganar:

- required tags
- restricted-ssh
- cloudtrail-enabled
- drift/compliance continuo
- remediacion automatica con SSM

Cuando normalmente no gana:

- alerta por CPU alta
- evento puntual
- deteccion de amenazas

Preguntas relacionadas del banco:

- 7, 9, 15, 18

### EventBridge

Que es:

- enrutador de eventos
- responde "cuando pasa X, dispara Y"

Cuando suele ganar:

- PR creado o actualizado
- build terminado
- evento de Config hacia SNS o Lambda
- push a CodeCommit

Cuando normalmente no gana:

- compliance continuo por si solo
- monitoreo de metricas por umbral

Preguntas relacionadas:

- 8, 18, 68, 118

### CloudWatch

Que es:

- metricas, logs, alarmas, dashboards

Subpartes que debes separar:

- CloudWatch Logs -> almacenamiento y consulta de logs
- Metric filter -> convierte patrones de logs en metricas
- Alarm -> toma decision sobre una metrica

Cuando suele ganar:

- "los logs ya estan en CloudWatch Logs y quiero alertar"
- "quiero detectar la palabra Critical"
- "quiero rollback con alarmas"

Preguntas relacionadas:

- 3, 4, 11, 16, 44, 100

### SNS

Que es:

- servicio de notificacion

Regla mental de examen:

- SNS distribuye
- SNS no debe cargar el filtro principal si ya conoces el evento exacto

Preguntas relacionadas:

- 3, 8, 16, 18

### Trusted Advisor

Que es:

- check administrado de mejores practicas

Cuando gana:

- menor esfuerzo de desarrollo
- underutilized EC2
- recomendacion ya existente de AWS

Cuando no gana:

- enforcement fino de tags
- compliance continuo de configuracion

Preguntas relacionadas:

- 85 y contraste con 15

### Systems Manager Automation

Que es:

- automatizacion y remediacion

Cuando gana:

- autoremediacion de Config
- pasos con aprobacion manual
- patching, runbooks, workflow operativo

Preguntas relacionadas:

- 7, 15, 18, 85

### CodeDeploy

Que es:

- servicio de deployment controlado

Debes dominar:

- Blue/Green
- canary
- deployment groups
- hooks
- rollback con alarmas

Trampas:

- confundirlo con Elastic Beanstalk
- confundir metricas con alarmas
- no distinguir BeforeAllowTraffic

Preguntas relacionadas:

- 5, 10, 19, 25, 58, 72, 93, 97

### Elastic Beanstalk

Que es:

- plataforma administrada para desplegar apps

Debes recordar:

- Blue/Green en Beanstalk = dos entornos + CNAME swap
- no es el mismo modelo de CodeDeploy

Preguntas relacionadas:

- 61, 62, 69, 75, 76

### CodePipeline / CodeBuild / CodeCommit

Debes separar:

- CodePipeline -> orquestacion
- CodeBuild -> build y test
- CodeCommit -> repo

Trampas:

- aprobacion manual no corrige un source branch mal elegido
- polling casi siempre pierde frente a eventos nativos
- PR automation correcta es event-driven

Preguntas relacionadas:

- 30, 36, 39, 63, 68, 72, 120

### Lambda concurrency

Provisioned concurrency:

- reduce cold starts

Reserved concurrency:

- reserva o limita capacidad
- no elimina cold starts

Preguntas relacionadas:

- 19, 54, 97, 107

### EKS cross-account

Patron que debes automatizar:

- AssumeRole
- permisos correctos en cuenta destino
- mapear rol en aws-auth

Preguntas relacionadas:

- 20

### CloudFormation (incluye StackSets)

Que es:

- base IaC declarativa para aprovisionar y actualizar recursos.

Debes dominar:

- Change sets
- DeletionPolicy (Retain, Snapshot)
- import de recursos
- drift detection y limites
- StackSets (multi-cuenta, multi-region)

Trampas:

- asumir que change set equivale a validacion funcional en runtime
- olvidar que custom resources tienen limites en drift detection

Preguntas relacionadas:

- 1, 2, 9, 22, 34, 35, 94

### IAM y STS (cross-account)

Debes separar:

- policy de permisos
- trust policy
- AssumeRole vs AssumeRoleWithSAML
- rol de servicio vs rol objetivo en cuenta destino

Regla mental de examen:

- si el escenario es cross-account deploy, casi siempre necesitas trust + AssumeRole + permisos en destino.

Preguntas relacionadas:

- 20, 36, 39, 43, 60, 98

### KMS y cifrado de artefactos

Debes dominar:

- key policy vs IAM policy
- grants
- cifrado de artefactos y buckets de pipeline
- uso cross-account de llaves

Trampa:

- pensar que cifrar un bucket resuelve por si solo permisos cross-account.

Preguntas relacionadas:

- 39, 43, 98

### S3 y logging

Debes separar:

- bucket policy
- server access logging
- eventos S3 para trigger
- versionado para despliegues (S3ObjectVersion)

Preguntas relacionadas:

- 4, 7, 45, 94, 104

### CloudTrail

Que es:

- auditoria de llamadas API y eventos de gestion/datos.

Regla mental de examen:

- CloudTrail da auditoria; Config da cumplimiento.

Preguntas relacionadas:

- 1, 12, 26, 119

### RDS y Aurora para DR

Debes dominar:

- read replica
- promotion de replica
- backup/restore vs replica
- tradeoff RTO/RPO

Preguntas relacionadas:

- 17, 41, 82

### ECS y ECR

Debes dominar:

- tagging/versionado de imagen
- task definition pinning (digest/tag)
- estrategia de deployment y consistencia de imagen

Preguntas relacionadas:

- 48, 107

### DynamoDB (cuando aparece en pipelines/eventos)

Uso tipico en banco:

- indice/metadata
- clave de particion y sort key alineadas al patron de consulta

Preguntas relacionadas:

- 104

### AWS Organizations y Control Tower

Debes dominar:

- cuenta de management vs cuentas miembro
- OUs
- patrones de despliegue centralizado desde cuenta DevOps
- StackSets con permisos service-managed

Trampas:

- confundir permisos de organizacion con permisos de despliegue cross-account
- asumir que por estar en la misma organizacion ya tienes acceso al recurso

Preguntas relacionadas:

- 1, 20, 21, 44, 65, 90

### Route 53 y decisiones de routing

Debes separar:

- failover routing
- latency-based routing
- weighted routing
- TTL como ajuste operativo, no solucion estructural

Trampa:

- pensar que bajar TTL resuelve por completo clientes que cachean mal DNS.

Preguntas relacionadas:

- 17, 84

### ALB y trafico en despliegues

Debes dominar:

- target groups
- health checks
- switching de trafico entre target groups

Trampa:

- cambiar DNS cuando el problema real se resuelve cambiando ruteo interno del ALB.

Preguntas relacionadas:

- 5, 25, 50, 84

### API Gateway y serverless de despliegue

Debes dominar:

- integracion con Lambda en pipelines
- gestion de cambios sin romper trafico
- distinguir cambios de app vs cambios de infraestructura

Preguntas relacionadas:

- 10, 54, 77, 116

### Step Functions (cuando aparece como distractor o coordinador)

Debes saber:

- coordina pasos y estado
- no reemplaza automaticamente monitoreo, compliance o deployment strategy

Regla mental:

- si la pregunta pide orquestacion de workflow multi-paso, puede ganar;
- si solo pide alerta o compliance simple, suele ser sobreingenieria.

Preguntas relacionadas:

- 11, 117

### GuardDuty vs Inspector

GuardDuty:

- deteccion de amenazas y comportamiento sospechoso

Inspector:

- vulnerabilidades y exposicion de workloads

Trampa de examen:

- usar GuardDuty para CVEs o Inspector para threat detection en tiempo real.

Preguntas relacionadas:

- 26, 70, 98, 110

### Secrets Manager vs Parameter Store

Secrets Manager:

- secretos sensibles y rotacion administrada

Parameter Store:

- configuracion y parametros, secure string con KMS cuando aplique

Trampa de examen:

- elegir Parameter Store cuando la pregunta exige rotacion nativa de secretos.

Preguntas relacionadas:

- 69, 98 (y refuerzo de temario oficial)

### Transit Gateway y arquitectura centralizada de egreso

Debes saber:

- TGW enruta trafico entre cuentas/VPCs
- logs de appliance y logs de red no son equivalentes

Trampa:

- mezclar Flow Logs de red con logs de aplicacion/firewall cuando el requisito es sobre severidad del firewall.

Preguntas relacionadas:

- 16, 21

## 2. Auto Scaling y DR que debes tener automaticos

### Scale in / scale out

- scale out = agregar capacidad
- scale in = quitar capacidad

### Spot vs On-Demand

- On-Demand = estable
- Spot = mas barato, interrumpible

### Mixed instances policy

- puede causar decisiones menos intuitivas en scale in

### Rebalancing

- puede exceder temporalmente el max capacity

Preguntas relacionadas:

- 14, 73

### DR strategies

Backup and Restore:

- mas barato
- mas lento

Pilot Light:

- mantienes lo minimo critico
- balance bueno costo/recuperacion

Warm Standby:

- copia reducida pero funcional
- mas rapido
- mas caro

Active/Active:

- dos entornos activos
- mas rapido
- mas caro

Preguntas relacionadas:

- 17, 82

## 3. Comparaciones que debes memorizar

### Config vs EventBridge

- Config = estado y compliance
- EventBridge = evento y reaccion

### CloudWatch metric vs CloudWatch alarm

- metric = dato
- alarm = decision

### Managed vs custom

- si AWS ya trae una opcion managed y cumple exacto el requisito, suele ganar

### Scheduled scaling vs target tracking

- horario predecible -> scheduled scaling
- metrica objetivo -> target tracking

### BeforeInstall vs BeforeAllowTraffic

- BeforeInstall = antes de instalar
- BeforeAllowTraffic = antes de trafico real

### Pilot Light vs Warm Standby

- Pilot Light suele ser mas costo-efectivo
- Warm Standby = mas rapido

### CloudTrail vs Config

- CloudTrail se usa para saber quien hizo que y cuando
- Config se usa para evaluar si el recurso cumple o no cumple

### key policy vs IAM policy (KMS)

- key policy abre la puerta de la llave
- IAM policy concede permisos del principal
- en cross-account, ambas pueden ser necesarias

### GuardDuty vs CloudTrail vs Config

- GuardDuty = deteccion de amenazas
- CloudTrail se usa como evidencia y auditoria de API
- Config se usa para cumplimiento de configuracion

## 4. Lista corta de memoria dura

- Config = cumplimiento continuo
- EventBridge = evento puntual
- CloudWatch Logs + metric filter + alarm + SNS = alerta por patron en logs
- Trusted Advisor = check administrado ya hecho
- SSM Automation = remediacion y workflow operativo
- CodeDeploy rollback suele apoyarse en alarmas; no basta con hablar de metricas sin una alarma evaluable
- Beanstalk Blue/Green suele resolverse con dos entornos y `CNAME swap`
- si el requisito exige artefacto de `production`, debe salir de esa rama; una aprobacion manual sobre `master` no cambia el source real
- Provisioned concurrency es la respuesta tipica cuando el dolor principal son cold starts
- Pilot Light suele encajar cuando el examen pide buen costo con recuperacion en horas, pero siempre valida RTO/RPO concretos
- en cross-account EKS, el patron tipico es `AssumeRole` + permisos correctos + mapeo en `aws-auth`
- `Retain` + import es el patron tipico para renombrar o reorganizar stacks sin perder recursos gestionados
- `S3ObjectVersion` suele ser la forma correcta de forzar actualizacion cuando el codigo de Lambda viene de S3 en CloudFormation
- `CloudTrail` se usa para auditoria; `Config` para compliance continuo
- `Organizations` no elimina por si solo la necesidad de trust policy y permisos cross-account
- bajar el TTL en Route 53 no sustituye una estrategia correcta de failover o traffic shifting

## 5. Huecos del banco a memorizar

Estos temas no estan tan bien cubiertos en el banco, pero si pueden aparecer en el examen o como distractores.

### Observabilidad y performance

- `AWS X-Ray` = tracing distribuido para seguir una solicitud entre servicios
- `CloudWatch dashboards` = visualizacion operativa, no alerta por si sola
- `Contributor Insights` = encontrar top talkers o patrones agregados en logs y metrics
- `metric math` = combinar metricas para tomar decisiones mas finas

### Secrets y configuracion en runtime

- `Secrets Manager` = secretos sensibles, rotacion, integracion nativa
- `Parameter Store` = configuracion y parametros; puede guardar secretos, pero no gana si el foco es rotacion
- si el escenario pide rotacion automatica o secretos administrados, piensa primero en `Secrets Manager`

### ECS y despliegues

- en `ECS`, revisa siempre estrategia de deployment, health checks y traffic shifting
- `CodeDeploy` puede controlar blue/green en ECS; no todo es rolling update simple
- si el escenario habla de seguridad del despliegue, compara `canary`, `linear` y `all-at-once`

### IaC y herramientas cercanas

- `CDK` = define IaC con lenguajes; sintetiza a `CloudFormation`
- `CloudFormation` sigue siendo el motor real de despliegue
- `OpsWorks` es mas legado; suele aparecer como distractor

### Governance y multi-cuenta

- `CloudTrail organization trails` = auditoria centralizada para `Organizations`
- `Control Tower` y `Service Catalog` pueden aparecer como capa de governance, no como reemplazo de permisos finos
- multi-cuenta no elimina `trust policy`, `AssumeRole` ni permisos del recurso

### Systems Manager mas alla de lo basico

- `Patch Manager` = parcheo administrado
- `State Manager` = mantener configuracion deseada en instancias
- `Automation` = runbooks y workflows operativos

### Red, edge y web estatico

- `Route 53` routing policies:
  - `failover` = primario/secundario
  - `weighted` = reparto porcentual
  - `latency` = menor latencia
  - `geolocation` = por ubicacion del usuario
- `CloudFront invalidation` = refrescar contenido cacheado en despliegues web estaticos

### Seguridad aplicada a pipelines

- revisar siempre:
  - permisos minimos de `CodeBuild`, `CodePipeline`, `CodeDeploy`
  - `artifact bucket`
  - cifrado con `KMS`
  - `key policy` y permisos cross-account

### Resiliencia y datos

- comparar siempre:
  - `backup/restore`
  - `pilot light`
  - `warm standby`
  - `active/active`
- fijate en:
  - lag de replicacion
  - tradeoff real de `RPO`
  - costo de mantener capacidad lista

## 7. Decision rapida en examen (20 segundos)

Si una pregunta es larga, no leas todo de corrido. Haz este filtro:

1. Detecta la palabra de resultado:

- compliance
- alerta
- deployment
- rollback
- DR
- cross-account
- branch/source

2. Detecta la restriccion dura:

- menor costo
- menor esfuerzo
- tiempo real
- sin downtime
- RTO/RPO

3. Mapea a servicio ganador:

- compliance continuo -> AWS Config
- evento puntual -> EventBridge
- alerta por logs -> CloudWatch Logs + metric filter + alarm + SNS
- deployment controlado -> CodeDeploy
- DR costo-efectivo con horas -> suele apuntar a `Pilot Light`, pero valida siempre RTO/RPO concretos
- cross-account cluster -> suele apuntar a `AssumeRole` + permisos + `aws-auth`

4. Elimina distractores por sobreingenieria:

- polling cuando ya hay eventos
- custom cuando ya hay managed
- scope mas amplio que el requisito

## 8. Checklist anti-error (tu perfil)

Antes de marcar opcion, valida estas 6 preguntas:

1. Estoy filtrando en la capa correcta o demasiado tarde?
2. La opcion usa el recurso exacto o uno parecido?
3. La opcion usa el estado exacto (NON_COMPLIANT vs ERROR)?
4. La solucion es managed cuando existe, o me fui a custom por reflejo?
5. La fuente/branch del despliegue es realmente la correcta?
6. Estoy confundiendo servicio correcto con estrategia correcta?
